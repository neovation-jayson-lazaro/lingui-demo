# Lingui Patterns, Setup & CLI Guide

This section is a developer guide covering how the app implements Lingui-based localization with the Next.js App Router. It explains the build pipeline, initialization patterns, navigation helpers, server-only utilities, middleware integration, static rendering safety, and CLI workflow ‚Äî with code references to the actual files in this repository.

---

## G1. Lingui Setup & Build Pipeline

### Configuration files

Six files control the Lingui + Next.js integration:

- **`lingui.config.ts`** ‚Äî single source of truth for locales, source locale, and catalog paths. Exports the `locales` tuple and `Locale` type directly. Imported by both the Lingui CLI and runtime code.
- **`babel.config.js`** ‚Äî declares the `next/babel` preset and `@lingui/babel-plugin-lingui-macro`.
- **`next.config.ts`** ‚Äî enables `reactCompiler: true` and `typedRoutes: true`.
- **`package.json`** ‚Äî contains `lingui:extract`, `lingui:compile`, `lingui:verify`, and `routes` scripts.
- **`tsconfig.json`** ‚Äî `@/*` path alias maps to `./src/*`. `.next/types` is in `include` for typed route generation.
- **`scripts/generate-routes.mts`** ‚Äî scans `src/app/[lang]/` for `page.tsx` files and generates `src/lib/routes.ts` with a typed `AppRoute` union.

### How the build pipeline works

This app uses **Babel** for source transformation. The reason: React Compiler requires a Babel plugin (`babel-plugin-react-compiler`), and Lingui's macro transformation (`@lingui/babel-plugin-lingui-macro`) must run in the same pipeline.

The pipeline for every `.ts`/`.tsx` file:

1. `next/babel` preset ‚Äî parses TypeScript and JSX
2. `@lingui/babel-plugin-lingui-macro` ‚Äî transforms `<Trans>` macros from `@lingui/react/macro` into runtime `i18n._()` calls
3. `babel-plugin-react-compiler` ‚Äî injected automatically by Next.js via `reactCompiler: true`

```js
// babel.config.js
module.exports = {
  presets: ["next/babel"],
  plugins: ["@lingui/babel-plugin-lingui-macro"],
};
```

```ts
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  reactCompiler: true,
  typedRoutes: true,
};

export default nextConfig;
```

The Lingui CLI (`lingui extract`, `lingui compile`) operates **independently** of the Babel build pipeline. The CLI reads source files to find `<Trans>` strings and manages `.po` / compiled `.ts` catalogs. The Babel plugin only handles runtime macro transformation ‚Äî it does not generate catalogs.

### Lingui config

```ts
// lingui.config.ts
import type { LinguiConfig } from "@lingui/conf";

export const locales = ["en", "fr", "es", "de"] as const;
export type Locale = (typeof locales)[number];

const config: LinguiConfig = {
  locales: [...locales],
  sourceLocale: "en",
  catalogs: [
    {
      path: "src/locales/{locale}",
      include: ["src/"],
    },
  ],
};

export default config;
```

`locales` defines the supported languages as a `const` tuple ‚Äî this enables the `Locale` type to be a string union (`"en" | "fr" | "es" | "de"`) rather than a loose `string`. `sourceLocale` tells the CLI that English `msgid` strings are the source text. `catalogs[0].include` tells the CLI to scan all of `src/` for `<Trans>` strings. `{locale}` in `path` is replaced with each locale code, producing `src/locales/en.po`, `src/locales/fr.po`, etc.

### Route generation

`scripts/generate-routes.mts` scans `src/app/[lang]/` for `page.tsx` files and writes `src/lib/routes.ts`:

```ts
// src/lib/routes.ts (auto-generated)
export const appRoutes = [
  "/",
  "/hellotesting",
  "/hellotesting/go-home"
] as const;

export type AppRoute = (typeof appRoutes)[number];
```

`AppRoute` is the union type used by all navigation helpers (`LocaleLink`, `useLocaleRouter`, `localeRedirect`) as their `href`/`path` parameter type. This provides compile-time route checking at the application level, independent of Next.js `typedRoutes`. Run `npm run routes` after adding or removing pages.

### TypeScript recognition of Lingui exports

Lingui's macro package (`@lingui/macro`) re-exports `<Trans>` as a compile-time-only construct. At build time, the Babel plugin replaces it with a call to `@lingui/react`'s runtime API. TypeScript sees the types from `@lingui/react/macro` (or `@lingui/macro`), which provide autocompletion and type checking for `<Trans>`, `useLingui`, and other Lingui APIs.

No special `tsconfig.json` settings are needed beyond the standard `@/*` path alias. The `.next/types/**/*.ts` entry in `include` is for Next.js typed routes, not Lingui.

### How catalogs are loaded at runtime

Catalogs are dynamically imported per locale in `src/lib/i18n.ts` via `import(`../locales/${locale}.ts`)`. The locale parameter is validated against the whitelist before any `import()` call ‚Äî preventing path injection. Loaded `Messages` objects are cached in a module-level `Map` because they are immutable. The `I18n` instances that use these messages are scoped per-request via `React.cache` ‚Äî see section G2.

## G2. Server-Side i18n Initialization (`setI18n`)

### Why `setI18n` is needed in every Server Component

Lingui's RSC integration does not use React context on the server. Instead, it stores the active `I18n` instance in a `React.cache`-scoped slot. This means **every Server Component that renders `<Trans>` must call `setI18n()` (or a wrapper that calls it) before rendering**. The layout's `setI18n` call does not propagate to child page components ‚Äî each page needs its own call.

### The server-only i18n module

`src/lib/i18n.ts` is the server-side i18n entry point. It is guarded by `import "server-only"`, which causes a build error if any client component tries to import it.

```ts
// src/lib/i18n.ts
import "server-only";

import type { Route } from "next";
import { cache } from "react";
import { redirect } from "next/navigation";
import { type I18n, type Messages, setupI18n } from "@lingui/core";
import { setI18n } from "@lingui/react/server";
import { locales, type Locale } from "../../lingui.config";
import type { AppRoute } from "./routes";

const catalogCache = new Map<string, Messages>();

async function loadCatalog(locale: string): Promise<Messages> {
  if (catalogCache.has(locale)) return catalogCache.get(locale)!;
  const { messages } = await import(`../locales/${locale}.ts`);
  catalogCache.set(locale, messages);
  return messages;
}

export function getAllLocales(): readonly Locale[] {
  return locales;
}

export function isLocale(value: string): value is Locale {
  return (locales as readonly string[]).includes(value);
}

const getI18nInstanceCached = cache(async (locale: string): Promise<I18n> => {
  const messages = await loadCatalog(locale);
  return setupI18n({ locale, messages: { [locale]: messages } });
});

export async function getI18nInstance(locale: string): Promise<I18n> {
  if (!(locales as readonly string[]).includes(locale)) {
    console.warn(`Unsupported locale "${locale}", falling back to "en"`);
    locale = "en";
  }
  return getI18nInstanceCached(locale);
}

export async function activateI18n(lang: string) {
  const i18n = await getI18nInstance(lang);
  setI18n(i18n);
  return { lang, i18n };
}

export function localeRedirect(lang: Locale, path: AppRoute): never {
  const suffix = path === "/" ? "" : path;
  redirect(`/${lang}${suffix}` as Route);
}
```

Key design decisions:

- **`catalogCache` (module-level `Map`)** ‚Äî holds immutable `Messages` data loaded from compiled catalogs. Safe to share across requests because the data is never mutated.
- **`getI18nInstanceCached` (wrapped in `React.cache`)** ‚Äî creates a mutable `I18n` instance scoped to the current server request. Each concurrent request gets its own instance. No state leaks between requests.
- **`activateI18n`** ‚Äî convenience helper that combines `getI18nInstance` + `setI18n` into one call, reducing boilerplate in pages.
- **`isLocale`** ‚Äî type guard that narrows `string` to `Locale`, used by the layout for validation.
- **`localeRedirect`** ‚Äî accepts typed `Locale` and `AppRoute` parameters, preventing invalid locale or route values at compile time. The `as Route` cast on the final string is safe because the `[lang]` dynamic segment makes any `/{slug}{suffix}` a valid `DynamicRoute` in the generated type.

### How the layout initializes i18n

`src/app/[lang]/layout.tsx` calls `getI18nInstance` and `setI18n` directly (not `activateI18n`) because it also needs the `i18n` instance to pass `i18n.messages` to `LinguiClientProvider`:

```tsx
// src/app/[lang]/layout.tsx (relevant excerpt)
import { setI18n } from "@lingui/react/server";
import { getI18nInstance, isLocale } from "@/lib/i18n";
import { LinguiClientProvider } from "@/components/LinguiClientProvider";

export default async function LangLayout({ params, children }: Props) {
  const { lang } = await params;
  if (!isLocale(lang)) notFound();

  const i18n = await getI18nInstance(lang);
  setI18n(i18n);

  return (
    <html lang={lang}>
      <body className={/* ... */}>
        <LinguiClientProvider
          initialLocale={lang}
          initialMessages={i18n.messages}
        >
          {children}
        </LinguiClientProvider>
      </body>
    </html>
  );
}
```

### How pages initialize i18n

Every page under `[lang]/` calls `activateI18n` at the top of its render function:

```tsx
// src/app/[lang]/page.tsx (relevant excerpt)
import { Trans } from "@lingui/react/macro";
import { activateI18n } from "@/lib/i18n";

export default async function Home({ params }: Props) {
  await activateI18n((await params).lang);

  return (
    <h1><Trans>Hello World</Trans></h1>
    // ...
  );
}
```

If you add a new page under `[lang]/`, you **must** call `activateI18n` (or manually call `getI18nInstance` + `setI18n`) before any `<Trans>` usage. Forgetting this will cause `<Trans>` to render empty strings or the raw message ID.

### How the I18n instance reaches client components

The server-side `I18n` object is **not serializable** ‚Äî it contains methods, closures, and internal state. It cannot be passed as a React prop across the server/client boundary. Instead, the layout passes two serializable values to `LinguiClientProvider`:

- `initialLocale` ‚Äî the locale string (e.g. `"fr"`)
- `initialMessages` ‚Äî the raw `Messages` object (`i18n.messages`)

The client provider reconstructs its own `I18n` instance from these ‚Äî see section G3.

## G3. Client Provider Patterns

### The client provider component

`src/components/LinguiClientProvider.tsx` is the client-side counterpart to `setI18n` on the server. It wraps the component tree with Lingui's `I18nProvider`, which uses React context to deliver the `I18n` instance to all client components.

```tsx
// src/components/LinguiClientProvider.tsx
"use client";

import { I18nProvider } from "@lingui/react";
import { type Messages, setupI18n } from "@lingui/core";
import { useMemo } from "react";

export function LinguiClientProvider({
  children,
  initialLocale,
  initialMessages,
}: {
  children: React.ReactNode;
  initialLocale: string;
  initialMessages: Messages;
}) {
  const i18n = useMemo(() => {
    return setupI18n({
      locale: initialLocale,
      messages: { [initialLocale]: initialMessages },
    });
  }, [initialLocale, initialMessages]);

  return <I18nProvider i18n={i18n}>{children}</I18nProvider>;
}
```

Key points:

- **`useMemo` prevents repeated initialization.** The `I18n` instance is only reconstructed when `initialLocale` or `initialMessages` changes ‚Äî i.e. when the user switches languages via navigation.
- **No runtime fetch.** The catalog data arrives via RSC serialization as part of the server-rendered response. The client does not need to fetch translation files separately.
- **Hydration is safe.** The server renders translated HTML using `setI18n` + `<Trans>`. The client hydrates with the same messages via `I18nProvider`. Because both use the same catalog data for the same locale, there are no hydration mismatches.

### How client components use translations

Client components use `<Trans>` from `@lingui/react/macro` directly. They inherit the `I18n` instance from `I18nProvider` via React context ‚Äî no additional setup is needed:

```tsx
// src/components/AdditionalTextToggle.tsx
"use client";

import { useState } from "react";
import { Trans } from "@lingui/react/macro";

export function AdditionalTextToggle() {
  const [enabled, setEnabled] = useState(false);
  return (
    <>
      <p><Trans>Additional text:</Trans> {/* toggle button */}</p>
      {enabled && <p><Trans>This is additional text.</Trans></p>}
    </>
  );
}
```

## G4. Typed Routes & Locale-Aware Navigation

### Two layers of route typing

The app has two complementary route typing mechanisms:

1. **Next.js `typedRoutes`** ‚Äî generates `.next/types/routes.d.ts` which constrains `<Link href>` and `router.push()` to known route strings like `"/[lang]"`, `"/[lang]/hellotesting"`, `"/[lang]/hellotesting/go-home"`. These include the `[lang]` dynamic segment.

2. **`AppRoute` union** ‚Äî generated by `scripts/generate-routes.mts` into `src/lib/routes.ts`. This produces locale-agnostic route strings like `"/"`, `"/hellotesting"`, `"/hellotesting/go-home"`. All navigation helpers (`LocaleLink`, `useLocaleRouter`, `localeRedirect`) accept `AppRoute` as their path parameter.

Pages pass locale-agnostic `AppRoute` values (e.g. `"/hellotesting"`) to navigation helpers, which prepend the locale and cast to `Route`. The `as Route` cast is safe because the `[lang]` dynamic segment makes `/${locale}${suffix}` match the `DynamicRoute` pattern in the generated types.

### Where `as Route` casts appear

Because locale-prefixed paths are constructed at runtime (the locale is a variable, not a literal), TypeScript cannot statically verify them against the generated `Route` type. All `as Route` casts are concentrated in **four files**:

- `src/components/LocaleLink.tsx` ‚Äî declarative link
- `src/components/LanguageSwitcher.tsx` ‚Äî language switch navigation
- `src/hooks/locale-hooks.ts` ‚Äî imperative navigation (`push`, `replace`, `prefetch`)
- `src/lib/i18n.ts` ‚Äî server-side `localeRedirect`

No page or application component ever constructs a locale-prefixed path manually ‚Äî they all go through these primitives using `AppRoute` values.

### Locale-aware link (`LocaleLink`)

`src/components/LocaleLink.tsx` wraps `next/link` and auto-prefixes `href` with the current locale:

```tsx
// src/components/LocaleLink.tsx
"use client";

import type { Route } from "next";
import Link, { type LinkProps } from "next/link";
import type { AppRoute } from "@/lib/routes";
import { useLocalePathname } from "@/hooks/locale-hooks";

type Props = Omit<LinkProps<Route>, "href"> & {
  href: AppRoute;
  children: React.ReactNode;
  className?: string;
};

export function LocaleLink({ href, ...props }: Props) {
  const { locale } = useLocalePathname();
  const suffix = href === "/" ? "" : href;
  const localizedHref = `/${locale}${suffix}` as Route;
  return <Link href={localizedHref} {...props} />;
}
```

Usage in pages:

```tsx
<LocaleLink href="/hellotesting">Go to Hello Testing</LocaleLink>
// When locale is "fr", renders: <Link href="/fr/hellotesting">
```

The `href` prop is typed as `AppRoute`, so only routes that exist in `src/lib/routes.ts` are accepted at compile time.

### Locale-aware pathname and router hooks

`src/hooks/locale-hooks.ts` provides two hooks for client components:

```ts
// src/hooks/locale-hooks.ts
"use client";

import type { Route } from "next";
import { usePathname, useRouter } from "next/navigation";
import { locales, type Locale } from "../../lingui.config";
import type { AppRoute } from "@/lib/routes";

function isLocale(value: string): value is Locale {
  return (locales as readonly string[]).includes(value);
}

export function useLocalePathname() {
  const pathname = usePathname();
  const [, first, ...rest] = pathname.split("/");
  const candidate = first ?? "";
  const locale: Locale = isLocale(candidate) ? candidate : "en";
  const pathnameWithoutLocale = (rest.length ? `/${rest.join("/")}` : "/") as AppRoute;
  return { pathname, locale, pathnameWithoutLocale } as const;
}

export function useLocaleRouter() {
  const router = useRouter();
  const { locale } = useLocalePathname();

  type NavOptions = Parameters<typeof router.push>[1];

  function toLocaleRoute(path: AppRoute): Route {
    const suffix = path === "/" ? "" : path;
    return `/${locale}${suffix}` as Route;
  }

  function push(path: AppRoute, options?: NavOptions) {
    router.push(toLocaleRoute(path), options);
  }

  function replace(path: AppRoute, options?: NavOptions) {
    router.replace(toLocaleRoute(path), options);
  }

  function prefetch(path: AppRoute) {
    router.prefetch(toLocaleRoute(path));
  }

  return { ...router, push, replace, prefetch, locale };
}
```

**`useLocalePathname()`** ‚Äî returns the full `pathname`, the validated `locale` (narrowed to `Locale` type via type guard), and the `pathnameWithoutLocale` (cast to `AppRoute`). Used by `LocaleLink` and `LanguageSwitcher`.

**`useLocaleRouter()`** ‚Äî wraps `useRouter()` with `push`, `replace`, and `prefetch` that accept `AppRoute` and auto-prefix the locale. The `NavOptions` type is correctly extracted from the underlying router. Use this for imperative client-side navigation:

```tsx
const router = useLocaleRouter();
router.push("/hellotesting");
// When locale is "es", calls: router.push("/es/hellotesting")
```

### Language switching

`src/components/LanguageSwitcher.tsx` switches languages by navigating to a new locale-prefixed URL ‚Äî not by dynamically swapping the locale at runtime. This is the correct pattern for App Router: it triggers a full server re-render with the new locale's layout and catalog, preventing stale server-rendered content.

The switcher reads the current `pathname` and `locale` from `useLocalePathname()`, then uses `useRouter()` directly from `next/navigation` to navigate. It deliberately does **not** use `useLocaleRouter` ‚Äî the switcher needs to *replace* the locale segment in the existing URL, not *prepend* the current locale to a route. Using `useLocaleRouter` here would double-prefix the locale.

The switcher preserves the current path, query parameters, and hash fragments:

```tsx
// src/components/LanguageSwitcher.tsx (relevant excerpt)
const { pathname, locale: currentLocale } = useLocalePathname();
const router = useRouter();
const segments = pathname.split("/");

const switchLocale = (newLocale: string) => {
  const newSegments = [...segments];
  newSegments[1] = newLocale;
  const newPath = newSegments.join("/");
  const { search, hash } = window.location;
  router.push(`${newPath}${search}${hash}` as Route);
};
```

Visiting `/en/hellotesting?foo=bar#section` and switching to French navigates to `/fr/hellotesting?foo=bar#section`.

## G5. Server-Only Redirects

### The `localeRedirect` utility

`src/lib/i18n.ts` exports a typed server-only redirect function:

```ts
export function localeRedirect(lang: Locale, path: AppRoute): never {
  const suffix = path === "/" ? "" : path;
  redirect(`/${lang}${suffix}` as Route);
}
```

The function accepts typed `Locale` and `AppRoute` parameters ‚Äî invalid locale strings or unknown routes are compile-time errors. Because it lives in a `"server-only"` module, it cannot be imported in client components. The return type is `never` ‚Äî like `notFound()`, it throws internally and halts rendering.

### Direct `redirect()` usage

The `go-home` page (`src/app/[lang]/hellotesting/go-home/page.tsx`) uses `redirect()` directly:

```tsx
import type { Route } from "next";
import { redirect } from "next/navigation";

export default async function GoHomePage({ params }: Props) {
  const { lang } = await params;
  redirect(`/${lang}` as Route);
}
```

This is a server-only page that immediately redirects to the locale root. The `as Route` cast is safe because `/${lang}` matches the `[lang]` dynamic route.

### Redirect loop safety

`localeRedirect(lang, path)` produces URLs like `/fr/login` which always have a valid locale prefix. The middleware recognizes valid locale prefixes and passes them through without redirecting. No redirect loop is possible between middleware and `localeRedirect`.

The middleware only redirects when **no** valid locale prefix is found. Once a valid prefix exists, the request passes through unconditionally.

## G6. Middleware Integration

### How middleware resolves the locale

`src/middleware.ts` intercepts every page navigation and ensures it reaches a locale-prefixed URL. The resolution chain:

1. **URL prefix** ‚Äî if the pathname starts with `/en/`, `/fr/`, `/es/`, or `/de/`, the request passes through. A `NEXT_LOCALE` cookie is set (1-year expiry) to remember the preference.
2. **Cookie** ‚Äî if no valid prefix is found, `getRequestLocale()` checks the `NEXT_LOCALE` cookie against the whitelist.
3. **Accept-Language** ‚Äî if no cookie exists, the `negotiator` library matches the browser's `Accept-Language` header against supported locales.
4. **Fallback** ‚Äî the first locale in `lingui.config.ts` (English) is used as the ultimate default.

```ts
// src/middleware.ts (resolution chain)
function getRequestLocale(request: NextRequest): string {
  const cookieLocale = request.cookies.get(LOCALE_COOKIE)?.value;
  if (cookieLocale && (locales as readonly string[]).includes(cookieLocale)) {
    return cookieLocale;
  }

  const langHeader = request.headers.get("accept-language") || undefined;
  const languages = new Negotiator({
    headers: { "accept-language": langHeader },
  }).languages(locales.slice());

  return languages[0] || locales[0] || "en";
}
```

### Invalid locale stripping

If the URL contains a locale-like prefix that isn't supported (e.g. `/pt/about`), the middleware strips it before redirecting:

```ts
const segments = pathname.split("/");
const firstPathSegment = segments[1];
const isLocaleFormat = /^[a-z]{2}(-[a-z]{2})?$/i.test(firstPathSegment ?? "");
const pathWithoutLocalePrefix = isLocaleFormat
  ? "/" + segments.slice(2).join("/")
  : pathname;
```

This prevents nested-junk URLs like `/en/pt/about`.

### Matcher

The middleware matcher excludes API routes, static assets, and images ‚Äî it only runs for page navigations:

```ts
export const config = {
  matcher: [
    "/((?!api|_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)",
  ],
};
```

### Three-layer validation summary

An invalid locale must pass through three layers before it could render:

1. **Middleware** ‚Äî redirects to a valid locale
2. **Layout** ‚Äî `if (!isLocale(lang)) notFound();`
3. **Catalog loader** ‚Äî `if (!(locales as readonly string[]).includes(locale))` falls back to `"en"` with a `console.warn`

All three layers read from the same `lingui.config.ts` `locales` tuple. There is no code path where an unsupported locale reaches a `<Trans>` component or an unvalidated dynamic import.

## G7. Module-Level Translation Safety

### The problem

When a page is statically rendered with `generateStaticParams`, Next.js calls the page component once per parameter combination. If a translation is resolved **at module evaluation time** (outside a component render function), it runs once when the module is first imported and freezes to whichever locale was active at that moment. All subsequent static renders for other locales receive the frozen value.

### What this looks like (do not do this)

```tsx
// BAD ‚Äî module-level translation freezes to a single locale
import { t } from "@lingui/macro";
import { getI18nInstance } from "@/lib/i18n";

const i18n = await getI18nInstance("en");
const pageTitle = t(i18n)`Hello World`; // frozen to English at import time

export default function Page() {
  return <h1>{pageTitle}</h1>; // always English, even for /fr and /es
}
```

### What this codebase does instead (correct)

All `<Trans>` usage occurs **inside component render functions**, after `activateI18n` or `setI18n` has been called:

```tsx
// CORRECT ‚Äî translation resolved during render, per-locale
import { Trans } from "@lingui/react/macro";
import { activateI18n } from "@/lib/i18n";

export default async function Page({ params }: Props) {
  await activateI18n((await params).lang);

  return <h1><Trans>Hello World</Trans></h1>;
}
```

This ensures `generateStaticParams` produces distinct HTML for each locale. No translation is resolved during module evaluation.

### Static rendering with `generateStaticParams`

The `[lang]` layout exports `generateStaticParams` to prerender all locale variants:

```tsx
// src/app/[lang]/layout.tsx
import { locales } from "../../../lingui.config";

export function generateStaticParams() {
  return locales.map((lang) => ({ lang }));
}
```

The build output is:

```
Route (app)
‚îú ‚óã /_not-found
‚îú ‚óè /[lang]                        ‚Üí /en, /fr, /es, /de
‚îú ‚óè /[lang]/hellotesting           ‚Üí /en/hellotesting, /fr/hellotesting, /es/hellotesting, /de/hellotesting
‚îî ‚óè /[lang]/hellotesting/go-home   ‚Üí /en/hellotesting/go-home, /fr/hellotesting/go-home, ... (server redirect)
```

Each rendered HTML file contains translations in the correct locale because `activateI18n` runs during each per-locale render pass ‚Äî not at import time.

# Production-Readiness Architecture Review

## 1. Architecture Summary

### Routing & Locale Segments

The app uses a single `[lang]` dynamic segment under `src/app/[lang]/` to partition all localized content. Three leaf pages exist:

- `page.tsx` ‚Äî home page
- `hellotesting/page.tsx` ‚Äî second page with navigation demos
- `hellotesting/go-home/page.tsx` ‚Äî server-only redirect page

Along with a translated 404 page (`not-found.tsx`) and a translated error boundary (`error.tsx`).

The root layout at `src/app/layout.tsx` is a pass-through ‚Äî it owns only global CSS imports and static `<Metadata>`. It returns `{children}` directly without wrapping in `<html>` or `<body>`. The `<html>` and `<body>` elements live in `src/app/[lang]/layout.tsx`, which has access to the `lang` parameter and renders `<html lang={lang}>` ‚Äî giving every locale an accurate document language attribute without forcing dynamic rendering.

`src/app/[lang]/layout.tsx` exports `generateStaticParams`, which reads the `locales` tuple directly from `lingui.config.ts` and emits one static build per supported locale. The expected build output is:

```
Route (app)
‚îú ‚óã /_not-found
‚îú ‚óè /[lang]                        ‚Üí /en, /fr, /es, /de
‚îú ‚óè /[lang]/hellotesting           ‚Üí /en/hellotesting, /fr/hellotesting, /es/hellotesting, /de/hellotesting
‚îî ‚óè /[lang]/hellotesting/go-home   ‚Üí /en/hellotesting/go-home, ... (server redirect)
```

### Route Typing & Next.js typedRoutes

The app maintains two complementary route typing mechanisms:

**Layer 1: Next.js typedRoutes.** `next.config.ts` enables `typedRoutes: true`, generating `.next/types/routes.d.ts` which constrains `<Link href>` and `router.push()` to known route strings including the `[lang]` segment (e.g. `"/[lang]/hellotesting"`).

**Layer 2: AppRoute union.** `scripts/generate-routes.mts` scans `src/app/[lang]/` for `page.tsx` files and generates `src/lib/routes.ts` with a `const` tuple and union type:

```ts
export const appRoutes = ["/", "/hellotesting", "/hellotesting/go-home"] as const;
export type AppRoute = (typeof appRoutes)[number];
```

All navigation helpers accept `AppRoute` as their path parameter. This provides compile-time route validation at the application level ‚Äî developers cannot pass a nonexistent route to `LocaleLink`, `useLocaleRouter`, or `localeRedirect` without a TypeScript error.

The `as Route` cast appears in four files that bridge between `AppRoute` and the Next.js `Route` type: `LocaleLink.tsx`, `LanguageSwitcher.tsx`, `locale-hooks.ts`, and `i18n.ts`. The cast is safe because the `[lang]` dynamic segment makes `/${locale}${suffix}` match the `DynamicRoute` pattern. No page or application component constructs locale-prefixed paths directly.

The `go-home/page.tsx` uses `redirect(`/${lang}` as Route)` directly ‚Äî this is a server-only page that immediately redirects, so the cast is appropriate and narrowly scoped.

### Locale Resolution

Locale resolution follows a three-tier priority chain inside `src/middleware.ts`:

1. **URL prefix** ‚Äî If the pathname starts with a known locale (`/en/...`, `/fr/...`, `/es/...`, `/de/...`), the request passes through. The middleware sets a `NEXT_LOCALE` cookie (1-year expiry, `sameSite: lax`) and injects an `x-locale` request header for potential downstream use.
2. **Cookie** ‚Äî If the URL has no valid locale prefix, `getRequestLocale()` checks the `NEXT_LOCALE` cookie against the whitelist.
3. **Accept-Language** ‚Äî If no cookie exists, the `negotiator` library matches the browser's `Accept-Language` header against supported locales.

The resolved locale is used to redirect the user to the correct prefixed URL (e.g. `/` ‚Üí `/en`).

### Invalid Locale Handling

Three layers of validation prevent unsupported locales from rendering:

- **Middleware layer** (`src/middleware.ts`, lines 46‚Äì51): Any pathname whose first segment matches `/^[a-z]{2}(-[a-z]{2})?$/i` but isn't in the whitelist gets stripped and redirected. For example, `/pt/about` ‚Üí `/en/about` (assuming English is the resolved preference). This prevents nested-junk URLs like `/en/pt/about`.
- **Layout layer** (`src/app/[lang]/layout.tsx`, line 39): `isLocale(lang)` type guard is checked and `notFound()` is triggered for any locale that somehow bypasses middleware (direct server-side navigation, cached route manifests, etc.).
- **Catalog loader layer** (`src/lib/i18n.ts`, lines 33‚Äì37): `getI18nInstance()` validates the locale parameter against the config whitelist before any `import()` call. Invalid locales fall back to `"en"` with a `console.warn`. This is defense-in-depth ‚Äî it should never be reached in normal operation.

### Translation Catalog Loading

`src/lib/i18n.ts` is guarded by `import "server-only"` ‚Äî it cannot be imported from client components. Catalogs are loaded lazily via dynamic `import(`../locales/${locale}.ts`)` and cached in a module-level `Map<string, Messages>`. This cache holds immutable `Messages` objects ‚Äî raw data from compiled catalogs ‚Äî which are safe to share across requests because they are never mutated after creation.

`I18n` instances (which are mutable) are scoped per-request via `React.cache`. Each request gets its own `I18n` instance constructed from the shared catalog data. This prevents mutable state from leaking between concurrent requests while still avoiding redundant dynamic imports and `JSON.parse` calls.

The compiled catalog files (`src/locales/en.ts`, `src/locales/fr.ts`, `src/locales/es.ts`, `src/locales/de.ts`) use `JSON.parse()` for the message payload. V8 parses `JSON.parse(string)` faster than equivalent JavaScript object literals ‚Äî this is the recommended Lingui pattern for performance.

No translations are resolved at module evaluation time. All `<Trans>` usage occurs inside component render functions, and `activateI18n` is called during rendering ‚Äî not at import time. This means no locale is "frozen" into a module during build, and `generateStaticParams` correctly produces distinct HTML for each locale.

### Lingui RSC Architecture (Server-Side i18n)

The `[lang]/layout.tsx` calls `getI18nInstance(lang)` followed by `setI18n(i18n)` from `@lingui/react/server` to establish the i18n context for layout-level `<Trans>` usage. Each page uses the `activateI18n()` convenience helper from `src/lib/i18n.ts`, which combines both calls into one:

```ts
await activateI18n((await params).lang);
```

This is required because Lingui's RSC integration stores the i18n instance in a `React.cache`-based scope ‚Äî each Server Component execution context needs its own `setI18n` call. The layout's call does not propagate to child page components. `<Trans>` from `@lingui/react/macro` is used for all translatable strings in Server Components.

The server-side `I18n` instance is never passed directly to client components ‚Äî only the serializable `locale` string and `messages` object are passed as props to `LinguiClientProvider`.

### Client Provider Architecture (Client-Side i18n)

`src/components/LinguiClientProvider.tsx` wraps children with Lingui's `I18nProvider`. It receives `initialLocale` and `initialMessages` as serialized props from the server layout, then constructs a client-side `I18n` instance via `useMemo`. The memo is keyed on both `initialLocale` and `initialMessages`, so the instance rebuilds when the user switches languages ‚Äî no stale state persists across locale changes.

The client receives the correct catalog data via RSC serialization at render time ‚Äî no additional runtime fetch is needed. Client components (`not-found.tsx`, `error.tsx`, `AdditionalTextToggle.tsx`, `ReturnHomeButton.tsx`) inherit i18n context from this provider without needing their own setup. They use `<Trans>` from `@lingui/react/macro` directly.

### Localized Navigation Hooks

`src/hooks/locale-hooks.ts` provides two client-side hooks that centralize locale-aware navigation:

- **`useLocalePathname()`** ‚Äî returns `{ pathname, locale, pathnameWithoutLocale }` parsed from `usePathname()`. The `locale` is validated via `isLocale()` type guard and narrowed to `Locale` type. The `pathnameWithoutLocale` is cast to `AppRoute`.
- **`useLocaleRouter()`** ‚Äî wraps `useRouter()` with `push`, `replace`, and `prefetch` methods that accept `AppRoute` and auto-prefix the locale. `NavOptions` type is extracted from the underlying router's `push` signature. A private `toLocaleRoute` helper handles the root-path edge case (`"/"` ‚Üí `"/${locale}"` without trailing slash).

Both hooks are `"use client"` ‚Äî they cannot be used in Server Components. The hooks correctly handle the edge case where the root path `"/"` should become `"/${locale}"` rather than `"/${locale}/"`.

### Locale-Aware Links

`src/components/LocaleLink.tsx` is a `"use client"` wrapper around Next.js `<Link>` that uses `useLocalePathname()` to auto-prefix the `href` with the current locale. Its `href` prop is typed as `AppRoute`, so only known routes are accepted at compile time. Pages use `<LocaleLink href="/hellotesting">` instead of manually interpolating `/${lang}/hellotesting`. The 404 page (`not-found.tsx`) also uses `LocaleLink` for locale-preserving navigation.

### Language Switching

`src/components/LanguageSwitcher.tsx` renders a `<select>` dropdown populated from the `locales` tuple in `lingui.config.ts`. On change, it uses `useLocalePathname()` to read the current path, swaps the locale segment, reads `window.location.search` and `window.location.hash` to preserve query parameters and hash fragments, and calls `router.push()` with the full reconstructed URL cast `as Route`. This triggers a full navigation to the new locale ‚Äî not a runtime locale swap ‚Äî ensuring the server renders fresh HTML with the correct catalog.

### Server-Side Redirects

`src/lib/i18n.ts` exports `localeRedirect(lang, path)` which accepts typed `Locale` and `AppRoute` parameters and calls `redirect()` from `next/navigation` with the locale prefix prepended. This is the server-side counterpart to the client-side `useLocaleRouter().push()`. Since it lives in a `"server-only"` module, it cannot be accidentally used in client components. The function's return type is `never` ‚Äî it throws internally like `notFound()`.

The `go-home` page demonstrates direct `redirect()` usage in a server-only context ‚Äî a page that exists solely to perform a locale-prefixed redirect.

### Middleware

`src/middleware.ts` uses the `middleware` file convention and handles all locale routing concerns:

- Locale-prefixed requests pass through with cookie persistence and header injection.
- Bare URLs (`/`, `/about`) get redirected to the preferred-locale-prefixed version.
- Unknown locale-like prefixes (`/pt/...` when `pt` is not supported) get stripped and redirected.
- The matcher excludes `/api`, `_next/static`, `_next/image`, `favicon.ico`, and common image extensions.

No redirect loops are possible: the middleware only redirects when no valid locale prefix is found. Once a valid prefix is present, the request passes through unconditionally. The `localeRedirect` server function and the `go-home` page both redirect to locale-prefixed paths, which the middleware recognizes and passes through ‚Äî no loop.

### Build Configuration

The build pipeline uses Babel instead of SWC to support both React Compiler and Lingui macro transformation:

- **`babel.config.js`** ‚Äî declares the `next/babel` preset (for TypeScript/JSX parsing) and `@lingui/babel-plugin-lingui-macro` (transforms `<Trans>` macros into runtime calls at compile time).
- **`next.config.ts`** ‚Äî enables `reactCompiler: true` (Next.js injects `babel-plugin-react-compiler` automatically) and `typedRoutes: true`.
- **`package.json`** ‚Äî includes `babel-plugin-react-compiler` and `@lingui/babel-plugin-lingui-macro` as dev dependencies.

The Lingui CLI tooling (`lingui:extract`, `lingui:compile`, `lingui:verify`) operates independently of the build pipeline ‚Äî it reads source files, extracts `<Trans>` strings into `.po` files, and compiles them into `.ts` catalogs. The Babel plugin only handles runtime transformation, not catalog generation.

### React Compiler Compatibility

All client components in the codebase are compatible with React Compiler:

- **`LinguiClientProvider`** ‚Äî uses `useMemo` with stable dependencies (`initialLocale`, `initialMessages`). React Compiler can optimize this further by auto-memoizing the render output.
- **`LanguageSwitcher`** ‚Äî reads from hooks (`useLocalePathname`, `useRouter`) and renders a controlled `<select>`. No side effects during render.
- **`LocaleLink`** ‚Äî reads from `useLocalePathname()` and renders `<Link>`. Pure render path.
- **`AdditionalTextToggle`** ‚Äî uses `useState` for a toggle. Pure render path.
- **`ReturnHomeButton`** ‚Äî reads from `useLocaleRouter()` and renders a `<button>` with an `onClick` handler. Pure render path.

`<Trans>` from `@lingui/react/macro` is transformed by the Babel plugin at compile time into a standard function call ‚Äî it does not introduce side effects or unstable references that would conflict with React Compiler assumptions. The macro output is a plain React element, which the compiler handles natively.

No component performs dynamic imports during render, mutates external state during render, or uses patterns that violate React Compiler's rules.

### 3.3 No `<head>` alternate hreflang tags

**Impact:** Search engines cannot discover alternate-language versions of a page from the HTML alone.   const { lang } = await params;
```tsx
  const alternates: Record<string, string> = {};
  for (const locale of locales) {
    alternates[locale] = `/${locale}`;
  }
  return {
    alternates: {
      languages: alternates,
    },
  };
}
```

### 3.4 Babel pipeline is slower than SWC

**Affected files:** `babel.config.js`, `next.config.ts`

The presence of `babel.config.js` disables SWC for source file transformation. All `.ts`/`.tsx` files are processed through Babel, which is significantly slower than SWC (typically 10‚Äì20x in raw transform speed). This is a necessary tradeoff: React Compiler requires Babel, and Lingui's Babel macro plugin must run in the same pipeline.

**Current impact:** Negligible for this codebase size. For large codebases (hundreds of files), build and HMR times may be noticeably slower. Monitor build times as the codebase grows.

**No fix available:** This is an inherent constraint of using React Compiler with Lingui. The SWC plugin for Lingui (`@lingui/swc-plugin`) cannot coexist with `reactCompiler: true` because they use incompatible compilation pipelines.

### 3.5 `pathnameWithoutLocale` cast to `AppRoute` is unchecked

**Affected file:** `src/hooks/locale-hooks.ts` (line 17)

```ts
const pathnameWithoutLocale = (rest.length ? `/${rest.join("/")}` : "/") as AppRoute;
```

The `as AppRoute` cast assumes the URL path (minus the locale prefix) corresponds to a known `AppRoute`. In practice, the App Router's 404 mechanism ensures only valid routes are served, so this cast is safe at runtime. However, if a custom catch-all route is added in the future, the cast could silently pass non-`AppRoute` values through the type system.

**Current impact:** None ‚Äî the 404 page handles unknown routes.

**Recommended fix:** No action needed currently. If catch-all routes are added, consider adding a runtime check or removing the cast.

### 3.6 `routes.ts` is generated but not validated in CI

**Affected files:** `scripts/generate-routes.mts`, `src/lib/routes.ts`

The route map in `src/lib/routes.ts` is auto-generated and committed to the repo. If a developer adds a new page but forgets to run `npm run routes`, the `AppRoute` type is stale ‚Äî but TypeScript will catch this at compile time because `LocaleLink` and `useLocaleRouter` only accept known `AppRoute` values. The developer gets a type error when trying to link to the new page.

**Current impact:** Low ‚Äî TypeScript catches the error. But a CI step would make the contract explicit.

**Recommended fix:** Add route generation to CI or to a pre-build script:

```json
{
  "scripts": {
    "prebuild": "npx tsx scripts/generate-routes.mts"
  }
}
```

## 4. Performance Assessment

### 4.1 Static generation confirmed

All locale pages are prerendered at build time via `generateStaticParams`. No route forces dynamic rendering. TTFB from a CDN edge is effectively zero ‚Äî the server delivers a static HTML file. `<html lang={lang}>` is baked into each prerendered HTML file with the correct locale value. With 4 locales and 3 pages (plus the redirect page), the build produces 12 static HTML files ‚Äî negligible build time.

### 4.2 Catalog size is negligible

Each compiled catalog (`en.ts`, `fr.ts`, `es.ts`, `de.ts`) is under 600 bytes. The RSC payload overhead from serializing `initialMessages` into `LinguiClientProvider` props is minimal. At this scale, there is no measurable bundle impact.

**When to act:** If a single compiled catalog exceeds ~50 KB (roughly 2,000‚Äì3,000 strings), split catalogs by route in `lingui.config.ts`:

```ts
catalogs: [
  { path: "src/locales/{locale}/common", include: ["src/components/"] },
  { path: "src/locales/{locale}/home", include: ["src/app/[lang]/page.tsx"] },
  { path: "src/locales/{locale}/hellotesting", include: ["src/app/[lang]/hellotesting/"] },
],
```

This gives each route only the strings it needs, reducing per-page RSC payload.

### 4.3 Catalog data is cached, I18n instances are request-scoped

`src/lib/i18n.ts` uses a module-level `Map<string, Messages>` to cache immutable catalog data. After the first `import()` for a locale, the raw messages are reused for all subsequent requests ‚Äî no repeated dynamic imports or `JSON.parse` calls.

`I18n` instances ‚Äî which are mutable and hold active locale state ‚Äî are created per-request via `React.cache`. This prevents cross-request state contamination while keeping the immutable data shared.

### 4.4 Client-side instance uses useMemo

`LinguiClientProvider` creates its `I18n` instance inside `useMemo`, keyed on `initialLocale` and `initialMessages`. It only reconstructs when the locale actually changes (i.e. language switch), not on every render. React Compiler may further optimize surrounding render paths.

### 4.5 No unnecessary client bundles

The `server-only` guard on `src/lib/i18n.ts` prevents the server catalog loader from being bundled into client JavaScript. Client components receive pre-resolved messages as serialized props ‚Äî they never import catalog files directly.

### 4.6 React Compiler reduces re-renders

With `reactCompiler: true`, React Compiler automatically memoizes component output and skips unnecessary re-renders. This reduces the performance impact of locale switching and interactive components like `AdditionalTextToggle` and `ReturnHomeButton` without manual `React.memo` wrappers.

## 6. Edge Cases & Security

### Cookie validation ‚Äî Safe

The `NEXT_LOCALE` cookie value is checked against the locale whitelist in `getRequestLocale()` (`src/middleware.ts`, line 72). An attacker setting `NEXT_LOCALE=../../etc/passwd` gets no match and falls through to `Accept-Language` negotiation. The cookie value never reaches a filesystem path or dynamic import without passing the whitelist check first.

### Dynamic import path injection ‚Äî Safe

`getI18nInstance()` validates the locale parameter against `(locales as readonly string[]).includes(locale)` before any `import()` call (`src/lib/i18n.ts`, line 34). An unsupported value is replaced with `"en"`. The `loadCatalog` function is module-private ‚Äî it cannot be called directly from outside `i18n.ts`. Even if it were, the `import(`../locales/${locale}.ts`)` path is constrained to the `src/locales/` directory by the relative path prefix.

### Route-level validation ‚Äî Safe

`[lang]/layout.tsx` line 39 calls `notFound()` for unknown locale segments via the `isLocale()` type guard. Visiting `/xyz/anything` produces a proper 404 response ‚Äî it does not render a page with an invalid locale.

### Open redirect via locale prefix stripping ‚Äî Safe

The middleware's locale-like prefix regex (`/^[a-z]{2}(-[a-z]{2})?$/i`) only matches two-letter codes with an optional region suffix. It strips the prefix and redirects to a same-origin URL constructed via `request.nextUrl` ‚Äî which preserves the host. There is no vector for open redirect or path traversal.

### x-locale header spoofing ‚Äî No impact

The middleware overwrites the `x-locale` header for valid-locale paths (line 27). No layout or page reads this header ‚Äî the `[lang]` layout uses the URL parameter directly. Spoofing `x-locale` in an inbound request has zero effect on rendered output.

### Unsupported locale in URL ‚Äî Safe

Triple-layer validation: middleware redirects to a known locale, the layout's `isLocale()` + `notFound()` catches any value that bypasses middleware, and the catalog loader falls back to `"en"` as final defense-in-depth. There is no code path where an unsupported locale reaches a `<Trans>` component or an unvalidated catalog import.

### Redirect loop between middleware, localeRedirect, and go-home page ‚Äî Safe

`localeRedirect(lang, path)` produces URLs like `/${lang}${path}` which always have a valid locale prefix. The `go-home` page produces `/${lang}` directly. The middleware recognizes valid locale prefixes and passes them through without redirecting. No loop is possible between any combination of these redirect sources.

### Query parameter preservation during language switch ‚Äî Safe

`LanguageSwitcher` reads `window.location.search` and `window.location.hash` and appends them to the new path. Query parameters and hash fragments survive locale changes.

### Module-level translation freeze ‚Äî Not present

No component or module calls `t(i18n)` or `<Trans>` at module evaluation time. All translation resolution occurs inside render functions, after `activateI18n` or `setI18n` has been called. This means `generateStaticParams` correctly produces distinct HTML per locale ‚Äî no locale is "frozen" into a shared module scope.

### AppRoute type safety boundary ‚Äî Safe

The `AppRoute` union is a closed set of known routes. Navigation helpers (`LocaleLink`, `useLocaleRouter`, `localeRedirect`) only accept `AppRoute` values, preventing construction of arbitrary paths. The `LanguageSwitcher` is the one navigation path that doesn't use `AppRoute` ‚Äî it manipulates the URL directly by swapping the locale segment, which is correct behavior since the rest of the path is already a known valid route.

## 7. Localized Navigation Summary

The codebase provides a complete set of locale-aware navigation primitives across server and client boundaries:

- **Server redirect (typed)** ‚Äî `localeRedirect(lang, "/path")` in `src/lib/i18n.ts`. Accepts `Locale` √ó `AppRoute`.
- **Server redirect (direct)** ‚Äî `redirect(`/${lang}` as Route)` in `src/app/[lang]/hellotesting/go-home/page.tsx`. For server-only redirect pages.
- **Client imperative navigation** ‚Äî `useLocaleRouter().push("/path")` in `src/hooks/locale-hooks.ts`. Accepts `AppRoute`.
- **Client declarative link** ‚Äî `<LocaleLink href="/path">` in `src/components/LocaleLink.tsx`. Accepts `AppRoute`.
- **Client pathname read** ‚Äî `useLocalePathname()` in `src/hooks/locale-hooks.ts`. Returns typed `Locale` and `pathnameWithoutLocale`.
- **Language switcher** ‚Äî `<LanguageSwitcher />` in `src/components/LanguageSwitcher.tsx`. URL segment manipulation with query/hash preservation.

All `as Route` casts are concentrated in four files (`LocaleLink.tsx`, `LanguageSwitcher.tsx`, `locale-hooks.ts`, `i18n.ts`). No page or application component ever constructs a locale-prefixed path manually.

## 10. Verdict

**The codebase is production-ready for Lingui-based localization.**

The architecture correctly implements every critical concern: locale-segmented routing with `[lang]` and `generateStaticParams` for full SSG across 4 locales, accurate `<html lang={lang}>` per locale without dynamic rendering, triple-layer locale validation (middleware ‚Üí layout `isLocale()` + `notFound()` ‚Üí catalog loader fallback), lazy-loaded catalogs with immutable data cached at module level and mutable `I18n` instances scoped per-request via `React.cache`, `server-only` guard preventing catalog code from leaking into client bundles, proper Lingui RSC architecture with `setI18n` in every Server Component scope and serializable-only props crossing the server/client boundary, `I18nProvider` with `useMemo`-based instance reconstruction on locale change, no module-level translation freezing, a complete set of typed locale-aware navigation primitives using `AppRoute` union at the API surface and narrowly-scoped `as Route` casts for Next.js compatibility (server redirect, client hooks, declarative links), cookie-based locale persistence with `Accept-Language` fallback, query/hash-preserving language switching via navigation (not runtime swap), server-only redirect pages, safe input validation at every layer, React Compiler compatibility with no side effects during render, and a Babel pipeline correctly chaining Lingui macro transformation with React Compiler optimization.
